# NexusEngine-KitUniverse Memory

Status: active

## Purpose

NexusEngine-KitUniverse is a new repository intended to hold a simple single-entry harness surface for running and validating sub-harness work against local model endpoints.

## Architecture Shape

- Harness design should prefer one simple entry point that dispatches to scoped sub-harnesses.
- Provider access must go through `ask_provider`; chains should not call LM Studio directly.
- Sub-harnesses should keep domain work separate from orchestration, gates, retries, and artifact ledgers.

## Major Conventions

- Keep repo-local operating state in `.agent/`.
- Keep durable repo decisions in this file when they are lasting and not merely turn notes.
- Use live endpoint probes before trusting remembered local-model availability.
- Treat destructive actions, credential use, deployments, and external submissions as approval-gated.
- KitUniverse count is a promotion target, not an attempt count. Failed candidates never advance it.
- Batch promotion requires fresh validation, four duplicate signatures, real NexusSimulator proof, Sol medium review, and a prepared/committed transaction journal.
- Batch resume reuses per-candidate `generation-results.jsonl`; completed matrix nodes must never be regenerated.
- RAWG source data remains external at `NexusRealtime-Ideas/games/rawg/chunks`; never copy it into KitUniverse runs.
- RAWG work is an editable live service. Code-hash epochs and atomic checkpoints replace repeated operator application builds.
- Every RAWG-derived candidate must retain its capability cluster, source IDs, evidence hash, and pipeline epoch through contract validation and promotion.
- Automatic writes stop at KitUniverse. ProtoKits and Nexus Engine integration always requires explicit approval.

## Implemented Harnesses

- `chain-ask-for-domain-list`: a three-step chain that seeds domain candidates, feeds them into an expansion/grouping step, then feeds that result into a final five-domain list.
- `workflow_harnesses/chainstormer/workflow_chainstormer.py`: a workflow harness that loops one model, feeds each prior input/output into the next iteration, and records every turn in a JSON list.
- `workflow_harnesses/dsk_domain_translator/workflow_dsk_domain_translator.py`: a workflow harness that translates an idea into a DSK work domain tree, with `--subdomains` controlling breadth and `--depth` controlling recursive layers.
- DSK translator defaults to `http://10.0.0.38:1234/v1` and `lfm2.5-350m-heretic-high-reasoning`, with gameplay matrix coverage recorded in `coverage-matrix.json`.
- `workflow_harnesses/domain_discovery/workflow_domain_discovery.py`: a looping workflow harness that takes an `IDEA SOURCE NAME` and description, passes accepted domains back as model memory, and asks for at most three new comma-separated domains per turn.
- Domain discovery defaults to `http://10.0.0.137:1234/v1` and `lfm2.5-1.2b-instruct`.
- Domain discovery needs rotating discovery lanes plus required source-phrase hints; otherwise `lfm2.5-1.2b-instruct` repeats early domains or drifts into app/plugin names.
- `workflow_harnesses/category_discovery/workflow_category_discovery.py`: a flat pre-hierarchy workflow that asks for categories associated with an input, records a category ledger, and writes `categories.json`.
- Category discovery should run before domain/subdomain hierarchy when the model starts inventing weak parent/child structures.
- `workflow_harnesses/subdomain_discovery/workflow_subdomain_discovery.py`: a ledger-backed workflow harness that loops through each seed domain and discovered subdomain, asks for up to three child subdomains per pass, records accepted/rejected names, and writes `subdomain-ledger.json` plus `subdomain-tree.json`.
- Subdomain discovery treats no-new-child turns as exhausted parent passes, not fatal failures, as long as the run discovers useful children overall.
- `workflow_harnesses/matrix_node_explorer/workflow_matrix_node_explorer.py`: MatrixNodeExplorer-Harness runs async matrix instances, chains short high-temperature generations, asks semantic `Y`/`N` game-connection checks, and writes records through sharded JSONL ingestion.
- `kituniverse_harness/smart_router.py`: smart routing service trims prompt context to a token budget and caps simultaneous provider predictions independently from matrix task concurrency.

## Guided Kit Builder

- `workflow_harnesses/guided_kit_builder/` converts full descriptions into comparable atomic kit contracts through structured model drafting, deterministic normalization, validation/repair, comparison, and link injection.
- Guided intake owns known facts; the model fills semantic gaps; deterministic contract logic owns identifiers, namespaces, boundary evidence, and safe fallback plans. Every final slot records its source.
- `SmartRoutingService(max_predictions=1)` is mandatory for this workflow even if the loaded LM Studio engine supports more parallel work.

## Ingestion Service

- Generated evidence under `runs/` is intentionally local and Git-ignored because production proofs can reach tens of gigabytes. Preserve or archive required run directories before deleting a checkout; commit source, contracts, compact fixtures, and proof summaries instead of raw run corpora.
- `kituniverse_harness/ingestion.py` provides a sharded JSONL ingestion service for high-concurrency dataset writes.
- Ingestion runs create `manifest.json`, `shards/*.jsonl`, `report.json`, and optional `stress-report.json`.
- Records are assigned to shards by stable hash of `record_id`; each shard has an async lock so 256 concurrent instances can write without sharing one SQLite/file lock.
